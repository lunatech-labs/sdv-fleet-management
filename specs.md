# Fleet Management Demo — Project Specification

**Version:** 1.0 (v1)
**Status:** Locked
**Audience:** External clients and prospects
**Team size:** 1 Engineer
**Timeline:** 2 Weeks

---

## 1. Overview

A client-facing demo showcasing Rust as a high-performance backend for vehicle fleet management, using Eclipse Kuksa to simulate a realistic fleet of 20 vehicles. The primary goal of v1 is to demonstrate how a Rust backend can retrieve and serve live vehicle data — identity, metadata, and real-time GPS position — flowing through an authentic automotive-grade signal pipeline.

The demo tells a realistic story: each vehicle runs its own Kuksa Databroker (mirroring the architecture of a real vehicle gateway), a sidecar bridge forwards VSS signals to a central MQTT broker, and the Rust backend aggregates and serves that data to a browser dashboard showing 20 vehicles moving on a live map.

---

## 2. Goals

### v1 must demonstrate
- One Kuksa Databroker per vehicle (realistic, per-vehicle signal ownership)
- VSS signals flowing from Kuksa → MQTT → Rust backend → browser
- A live Leaflet map with 20 vehicle pins updating in real time via WebSocket
- Per-vehicle detail: VIN, make, model, software version
- A clean REST + OpenAPI surface documenting the backend

### Out of scope for v1
- OTA update campaigns (reserved for v2, will use Eclipse HawkBit)
- Authentication / TLS (demo environment only)
- Persistent storage / database
- Historical telemetry replay
- More than 20 vehicles

## 3. Architecture

### 3.1 Component overview

```
Browser Dashboard
    │  REST GET /fleet (once, on load)
    │  WebSocket /ws/fleet (persistent, position push)
    ▼
Rust Backend (axum)
    │  MQTT subscribe: kuksa/+/telemetry/#
    ▼
Eclipse Mosquitto (MQTT broker, port 1883)
    ▲
    │  MQTT publish: 2 dynamic signals per vehicle at 1 Hz (lat/lon)
    │
┌───────────────────────────────────────────┐
│  Vehicle fleet — 20 instances             │
│                                           │
│  ┌─────────────────┐  ┌─────────────────┐ │
│  │ Kuksa Databroker│→ │ kuksa2mqtt      │ │
│  │ (gRPC, per port)│  │ sidecar         │ │
│  └─────────────────┘  └─────────────────┘ │
│           × 20 vehicles                   │
└───────────────────────────────────────────┘
    ▲
    │  Seed script (runs once at startup)
```

### 3.2 Per-vehicle unit

Each of the 20 vehicles is a self-contained pair of containers:

- **Kuksa Databroker** — holds that vehicle's VSS signal tree. Exposes a gRPC interface on a unique port (55556–55575).
- **kuksa2mqtt sidecar** — subscribes to its local Databroker over gRPC and republishes each VSS signal to Mosquitto under the topic tree defined in section 5. Also runs the GPS random walk logic for dynamic signals.

This mirrors how a real vehicle's gateway ECU would forward signals to a cloud broker.

### 3.3 Port allocation

| Service | Port |
|---|---|
| Mosquitto MQTT | 1883 |
| Kuksa Databroker VIN-0001 | 55556 |
| Kuksa Databroker VIN-0002 | 55557 |
| … | … |
| Kuksa Databroker VIN-0020 | 55575 |
| Rust backend | 3000 |
| Frontend dev server | 8080 |

## 4. VSS Signals

Six VSS signals are used per vehicle in v1.

Note: `Vehicle.SoftwareVersion` is a custom extension — no standard VSS equivalent exists in the current COVESA catalog.

| VSS path | Kind | Update rate | Description |
|---|---|---|---|
| `Vehicle.VehicleIdentification.VIN` | Static | Once (seed) | Vehicle identifier |
| `Vehicle.VehicleIdentification.Brand` | Static | Once (seed) | Manufacturer name |
| `Vehicle.VehicleIdentification.Model` | Static | Once (seed) | Vehicle model |
| `Vehicle.SoftwareVersion` | Static | Once (seed) | Current firmware version |
| `Vehicle.CurrentLocation.Latitude` | Dynamic | 1 Hz | GPS latitude, random walk |
| `Vehicle.CurrentLocation.Longitude` | Dynamic | 1 Hz | GPS longitude, random walk |

**Static signals** are written into each Databroker by the seed script at startup and do not change at runtime.

**Dynamic signals** are generated by the `kuksa2mqtt` sidecar. Each vehicle starts at a seeded latitude/longitude (spread across a configurable city bounding box). Every second, the sidecar applies a Gaussian delta (σ ≈ 0.0002°, ~20 metres) to both coordinates, writes the new values into its local Databroker, and the bridge picks them up and publishes to Mosquitto.

## 5. MQTT Topic Tree

The topic structure is designed for v1 clarity and forward compatibility with v2 OTA commands.

Note: The MQTT topic path mirrors the VSS signal path with the `Vehicle.` prefix stripped.

```
kuksa/{vin}/telemetry/VehicleIdentification/VIN
kuksa/{vin}/telemetry/VehicleIdentification/Brand
kuksa/{vin}/telemetry/VehicleIdentification/Model
kuksa/{vin}/telemetry/SoftwareVersion
kuksa/{vin}/telemetry/CurrentLocation/Latitude
kuksa/{vin}/telemetry/CurrentLocation/Longitude

kuksa/{vin}/ota/command     ← reserved, not used in v1
kuksa/{vin}/ota/status      ← reserved, not used in v1
```

The Rust backend subscribes once to `kuksa/+/telemetry/#`. The VIN is parsed from topic segment index 1. A single wildcard subscription covers all 20 vehicles and all future telemetry signals automatically.

**Payload format:** All signals are published as plain UTF-8 strings. Numeric values (latitude, longitude) are serialised as decimal strings (e.g. `"48.8566"`).

## 6. Rust Backend

### 6.1 Responsibilities

- Subscribe to Mosquitto on startup via `rumqttc`.
- Apply CORS middleware (`tower-http` `CorsLayer`) to allow browser requests from the frontend origin (`:8080`).
- Parse incoming MQTT messages: extract VIN from topic, signal name from trailing path, value from payload.
- Maintain an in-memory `DashMap<String, VehicleRecord>` as the live state store.
- On each position update: write to `DashMap` and send a `PositionEvent` onto a `tokio::sync::broadcast` channel.
- Serve REST endpoints for metadata queries.
- Serve a WebSocket endpoint that streams `PositionEvent`s to connected browsers.
- Expose OpenAPI documentation at `/docs` via Swagger UI.

### 6.2 Data model

```rust
struct VehicleRecord {
    vin:              String,
    brand:            String,
    model:            String,
    software_version: String,
    latitude:         f64,
    longitude:        f64,
    last_seen:        DateTime<Utc>,
}

struct PositionEvent {
    vin: String,
    lat: f64,
    lon: f64,
}
```

### 6.3 Internal data flow

```
rumqttc event loop
    → parse topic → extract VIN + signal
    → update DashMap<VIN, VehicleRecord>
    → if signal is Latitude or Longitude:
        broadcast_tx.send(PositionEvent { vin, lat, lon })

axum WebSocket handler (one per connected browser tab)
    → let rx = broadcast_tx.subscribe()
    → loop { ws.send(Json(rx.recv().await)) }
```

### 6.4 REST API

| Method | Path | Description |
|---|---|---|
| `GET` | `/fleet` | Returns all 20 vehicles with full `VehicleRecord` |
| `GET` | `/vehicles/{vin}` | Returns a single vehicle record |
| `GET` | `/health` | Liveness check, returns `200 OK` |
| `GET` | `/docs` | Swagger UI (OpenAPI) |
| `WS` | `/ws/fleet` | Persistent WebSocket, pushes `PositionEvent` JSON |

**`GET /fleet` response shape:**

```json
[
  {
    "vin": "VIN-0001",
    "brand": "Toyota",
    "model": "Camry",
    "software_version": "1.4.2",
    "latitude": 48.8566,
    "longitude": 2.3522,
    "last_seen": "2026-04-14T10:00:00Z"
  }
]
```

**WebSocket message shape (pushed at 1 Hz per vehicle):**

```json
{ "vin": "VIN-0001", "lat": 48.8571, "lon": 2.3529 }
```

### 6.5 Crate list

| Crate | Role |
|---|---|
| `axum` | REST API + WebSocket handlers |

## 7. Frontend

### 7.1 Technology

Vue 3 (Composition API).

**Key dependencies:**

| Library | Role |
|---|---|
| `leaflet` + `@vue-leaflet/vue-leaflet` | Interactive map with vehicle pins |
| `native WebSocket API` | Live position updates (no extra library needed) |

### 7.2 Behaviour

On page load:
1. Call `GET /fleet` once to render the full vehicle table and place initial map pins.
2. Open a persistent `WebSocket` to `ws://localhost:3000/ws/fleet`.
3. On each WebSocket message, call `marker.setLatLng([lat, lon])` on the relevant Leaflet marker. Leaflet handles smooth repositioning natively — no full re-render.

On pin click:
- Open a side drawer displaying the vehicle's VIN, brand, model, and software version from the initial REST payload (no additional request needed).

### 7.3 UI views

- **Map view** — full-screen Leaflet map, 20 vehicle pins labelled by VIN, moving in real time.
- **Vehicle detail drawer** — slides in on pin click; shows static metadata.
- **Fleet table** (optional, secondary tab) — tabular view of all 20 records for a data-oriented audience.

## 8. Seed Script

Runs once before the backend starts (enforced via `docker-compose` `depends_on` + `healthcheck`).

**Responsibilities:**
1. For each of the 20 vehicles, connect to its Databroker over gRPC.
2. Write the four static VSS signals (VIN, Brand, Model, SoftwareVersion).
3. Write initial Latitude and Longitude values within the configured city bounding box.
4. Exit with code 0 on success (docker-compose health gate).

**Implementation language:** Python (using `kuksa-client`).

## 9. docker-compose Orchestration

**Startup order:**

1. `mosquitto` — starts first, no dependencies.
2. `databroker-{01..20}` — 20 Kuksa Databroker containers, each on a unique port.
3. `seed` — runs after all databrokers are healthy; exits after writing all signals.
4. `kuksa2mqtt-{01..20}` — 20 sidecar containers, each pointed at one databroker and at Mosquitto; start after seed completes.
5. `backend` — starts after Mosquitto is healthy; begins MQTT subscription immediately.
6. `frontend` — starts last;

**Named services (excerpt):**

```yaml
services:
  mosquitto:
    image: eclipse-mosquitto:2
    ports: ["1883:1883"]

  databroker-01:
    image: ghcr.io/eclipse-kuksa/kuksa-databroker:latest
    ports: ["55556:55555"]

  kuksa2mqtt-01:
    build: ./kuksa2mqtt
    environment:
      KUKSA_HOST: databroker-01
      KUKSA_PORT: 55555
      MQTT_HOST: mosquitto
      VEHICLE_VIN: VIN-0001
    depends_on: [mosquitto, seed]

  backend:
    build: ./backend
    ports: ["3000:3000"]
    environment:
      MQTT_HOST: mosquitto
    depends_on: [mosquitto, seed]

  frontend:
    build: ./frontend
    ports: ["8080:8080"]
    depends_on: [backend]
```

## 11. v2 Roadmap (OTA Update Campaigns)

Here, we will explore usage of Eclipse HawkBit.

The topic tree and architecture are intentionally designed to support v2 without breaking changes.

**Reserved MQTT paths:**
```
kuksa/{vin}/ota/command    # backend → vehicle
kuksa/{vin}/ota/status     # vehicle → backend
```

**Planned v2 capabilities:**
- `POST /campaigns` — trigger an OTA update campaign across selected vehicles.
- `GET /campaigns/{id}` — poll campaign progress per vehicle.
- Per-vehicle state machine: `PENDING → DOWNLOADING → INSTALLING → COMPLETE / FAILED`.
- Configurable failure rate (e.g. 10% of vehicles fail) for realistic rollback demo.
- Dashboard campaign panel with per-vehicle progress bars.

**Open design decision for v2:** Whether the OTA state machine lives in the Rust backend (commands flow via MQTT, status flows back via MQTT — more realistic) or whether the backend writes directly to Kuksa VSS signals via gRPC (simpler but bypasses the MQTT layer for commands). The MQTT-native approach is recommended for architectural consistency.
